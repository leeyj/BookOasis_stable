# 구글 드라이브 기반 서버 간 도서 공유 플러그인 구성 계획서

## 1. 개요
BookOasis 미디어 서버 간에 기가바이트(GB) 단위의 대용량 시리즈 도서 파일을 빠르고 효율적으로 공유하기 위한 플러그인 아키텍처 계획서입니다. 
로컬 네트워크 대역폭 및 디스크 I/O 병목을 방지하기 위해 **구글 드라이브 간 서버사이드 복사(Server-side Copy)** 방식을 강제하는 것을 핵심 전략으로 삼습니다.

---

## 2. 핵심 아키텍처 및 원칙

* **구글 드라이브 한정:** 로컬 파일 다운로드는 배제합니다. 구글 드라이브가 연동된 사용자끼리만 공유가 가능하도록 제한하여 개발 복잡도를 낮추고 전송 속도를 극대화합니다.
* **서버사이드 복사 (Zero Local Traffic):** 파일 전송 시 로컬 서버의 대역폭을 전혀 사용하지 않고 구글 데이터센터 내부에서 즉시 복사되도록 Rclone 기능을 활용합니다.
* **마이크로 중계 서버 (Message Broker):** A 서버와 B 서버 간의 직접 통신이 어려운 환경(홈 네트워크, 방화벽)을 고려해 중간에 매우 가벼운 메시지 큐 역할의 중계 서버(Serverless)를 둡니다. 대용량 파일은 중계 서버를 거치지 않습니다.

---

## 3. 워크플로우 (Step-by-Step)

### [Step 1] A 사용자 (발신자)의 공유 요청
1. 웹 UI에서 특정 작품의 컨텍스트 메뉴를 열고 **"이 시리즈를 공유"** 버튼 클릭
2. 플러그인이 해당 작품의 부모 폴더(시리즈 전체)에 대해 구글 드라이브 공유 권한(링크가 있는 모든 사용자 보기)을 오픈
3. 플러그인이 중계 서버로 "공유 폴더 ID, 작품명, 발신자/수신자 ID"를 담아 Push 요청 (`POST /api/share/push`)

### [Step 2] B 사용자 (수신자)의 수신 대기 및 처리
1. B 서버의 플러그인은 백그라운드 스케줄러(또는 자체 루프)를 통해 주기적으로(예: 5분마다) 중계 서버에 Pull 요청 (`GET /api/share/pull`)
2. 자신에게 도착한 새로운 공유 항목이 발견되면 중계 서버에 즉시 Ack 응답 (`POST /api/share/ack`)을 보내 메시지를 큐에서 삭제
3. B 서버는 환경설정에 등록해둔 자신의 구글 드라이브 리모트 경로로 Rclone 서버사이드 복사 명령 실행

### [Step 3] Rclone 백그라운드 복사 및 DB 반영
1. B 서버 플러그인이 아래의 Rclone 명령을 백그라운드로 실행:
   ```bash
   rclone copy --drive-server-side-across-configs {A의_공유폴더_ID} {B의_gdrive_리모트}:{저장경로}/{작품명}
   ```
2. Rclone 복사가 완료되면(구글 서버 내에서 이루어지므로 매우 빠름), 해당 라이브러리의 VFS(가상 파일 시스템) 디렉토리 갱신(Refresh) 수행
3. BookOasis 스캐너를 트리거(`process_folder_task`)하여 수신받은 새 시리즈를 라이브러리 DB에 즉시 등록

## 4. 중계 서버 (Relay Server) 규격 제안

중계 서버는 상태를 길게 유지하지 않는 서버리스(Vercel, Cloudflare Workers, AWS Lambda 등)로 구축하여 유지비용을 최소화(무료)합니다.

### 4.1. `POST /api/share/push` (공유 등록)
- **주체:** A 서버 -> 중계 서버
- **역할:** 공유할 항목의 메타데이터를 중계 큐에 등록
- **Payload 예시:**
  ```json
  {
    "sender_id": "user-a",
    "target_id": "user-b",
    "folder_id": "1A2B3C4D5E6F7G8H9I0J",
    "series_name": "나루토",
    "auth_token": "secret-token"
  }
  ```

### 4.2. `GET /api/share/pull?target_id={user-b}` (알림 조회)
- **주체:** B 서버 -> 중계 서버
- **역할:** B 서버에게 온 새 공유 항목이 있는지 조회 (주기적 Polling)
- **Response 예시:**
  ```json
  {
    "success": true,
    "messages": [
      {
        "msg_id": 101,
        "folder_id": "1A2B3C4D5E6F7G8H9I0J",
        "series_name": "나루토",
        "created_at": "2026-07-10T23:05:00Z"
      }
    ]
  }
  ```

### 4.3. `POST /api/share/ack` (수신 확인)
- **주체:** B 서버 -> 중계 서버
- **역할:** 폴더 ID를 성공적으로 수신하여 큐에서 항목 제거
- **Payload 예시:**
  ```json
  {
    "target_id": "user-b",
    "msg_id": 101,
    "auth_token": "secret-token"
  }
  ```

---

## 5. 향후 확장성 및 고려사항
* **보안:** `auth_token`을 활용한 인증 체계를 도입하여 허가되지 않은 사용자의 접근 방지. 커뮤니티 형태로 확장 시 '초대 코드' 발급 시스템 도입 고려.
* **진행률 피드백:** 서버사이드 복사라 할지라도 기가바이트(GB) 규모일 경우 수십 초에서 몇 분이 걸릴 수 있음. Rclone 로그를 파싱하여 B 서버 웹 UI에 '다운로드 중' 알림을 표시할 수 있는 시스템 훅(Hook) 연동 고려 필요.
