# 📑 북 오아시스 핵심 기능 기술 명세서 (Core Feature Specifications)

이 문서는 북 오아시스(BookOasis) 미디어 서버가 자랑하는 압도적인 성능과 확장성을 가능하게 하는 핵심 기술 설계 및 아키텍처 구현 사양에 대해 설명합니다.

---

## 1. 오프셋(Offset) 기반 압축 파일 실시간 스트리밍 뷰어

### 💡 개요
전통적인 만화책 뷰어 서버들은 사용자 요청 시 ZIP/CBZ 압축 파일 전체를 서버 임시 폴더에 해제한 뒤 이미지를 제공하므로, 디스크 I/O 병목 및 메모리 낭비가 매우 심했습니다.
북 오아시스는 **압축 파일 전체 해제 없이** 필요한 이미지 바이트만 파일 채널에서 직접 슬라이싱해 전송하는 초고속 실시간 스트리밍 아키텍처를 채택했습니다.

### 🛠️ 구현 메커니즘
1. **스캔 단계**: 스캐너가 ZIP/CBZ 파일을 스캔할 때, 중앙 디렉터리 구조를 읽어 아카이브 내부 각 이미지 파일의 **로컬 헤더 물리 바이트 시작 오프셋(`local_header_offset`)**, **압축 크기(`compress_size`)**, **무압축 크기(`file_size`)**, **압축 방법**을 추출하여 `book_offsets` 테이블에 사전 인덱싱합니다.
2. **요청 단계**: 클라이언트 뷰어에서 특정 페이지를 요청하면 `/api/books/<id>/page/<page_num>` 라우터가 활성화됩니다.
3. **스트리밍 단계**:
   - 서버는 DB에서 해당 페이지의 오프셋 구조를 수 초 내에 찾아냅니다.
   - 원본 ZIP 파일을 열어 `f.seek(local_header_offset)`으로 즉각 이동합니다.
   - 정확히 해당 이미지에 해당하는 바이트 길이만큼만 스트림 형태로 읽어 즉각 메모리상에서 압축 해제(Decompress) 후 브라우저에 바이너리로 반환합니다.
   - **효과**: 1GB가 넘는 압축 파일도 단 10ms 이내에 첫 페이지 로딩이 완료되며 서버 메모리 점유가 극도로 억제됩니다.

## 2. Kavita 방식의 프론트엔드 최적화 (선로드 & 지연 로딩)

### 💡 개요
수만 권의 도서 목록을 스크롤할 때 브라우저 렌더링 프레임 저하 및 이미지 로딩 지연 현상(버퍼링)을 차단하고, 뷰어에서 초당 수십 장의 만화책 이미지를 대기 랙(Latency) 없이 0ms 즉시 스위칭하기 위한 렌더링 최적화 모델입니다.

### 🛠️ 구현 메커니즘
* **Intersection Observer 기반 지연 로딩 (Lazy Loading)**:
  - 뷰포트에 실제로 진입하기 전의 도서 카드 이미지 노드들은 빈 플레이스홀더로 유지되며, 진입 시점에만 썸네일을 요청하여 동시 HTTP 요청 커넥션 낭비를 예방합니다.
* **Blob URL 인메모리 캐싱 (In-Memory Blob URL Caching)**:
  - 이미지 태그의 `src` 속성을 서버 API URL로 지정할 경우 발생하는 브라우저 자체 지연을 회피하기 위해, JavaScript 비동기 `fetch` API를 이용해 이미지 데이터를 이진(Blob) 데이터로 받아 전역 캐시 맵(`blobCacheMap`)에 보관하고, `URL.createObjectURL(blob)`을 통해 브라우저 로컬 캐시 메모리에 즉각 스왑 렌더링합니다.
* **순차 백그라운드 프리로드 스케줄러 (Sequential Preload Queue)**:
  - 다수의 페이지를 동시에 HTTP 요청할 때 발생하는 대역폭 병목을 차단하기 위해, 한 번에 1장씩 순차적으로 다운로드하여 캐싱을 수행하는 백그라운드 큐 스케줄러(`startSequentialPreload`)를 구동합니다.
* **DocumentFragment를 이용한 돔(DOM) 렌더링 일괄 반영**:
  - 스크롤을 내릴 때마다 수십 개의 노드를 매번 개별 삽입하여 리플로우(Reflow)를 발생시키는 대신, 메모리상에서 `DocumentFragment` 객체로 목록 카드를 결합한 뒤 한 번에 DOM에 투입하여 60fps에 달하는 부드러운 그리드 스크롤을 실현합니다.

## 3. 동적 메타데이터 플러그인 아키텍처 및 설정 보존

### 💡 개요
북 오아시스는 외부 소스(알라딘 등)로부터 도서 메타데이터를 수급할 때 결합도를 낮추기 위한 플러그인 아키텍처를 도입했습니다.

### 🛠️ 구현 메커니즘
* **`MetadataPlugin` 베이스 클래스 제공**:
  - 공통 규격 메서드 `search_books()`, `get_book_detail()`을 선언하여 플러그인 구현을 표준화합니다.
* **동적 모듈 로드 및 설정 보존**:
  - `plugins/` 디렉터리에 정의된 플러그인 파일들이 서버 기동 시 자동으로 인덱싱되고, 관리자 페이지에 토글 버튼으로 나타납니다.
  - 플러그인별로 고유의 설정 스키마(`Config Schema`)를 정의할 수 있어, API Key 등을 안전하고 개별적인 인터페이스로 데이터베이스(`settings` 테이블)에 저장할 수 있습니다.
* **수동 메타데이터 선택 및 교체**:
  - 책 상세 화면에서 사용자가 '정보 찾기'를 수행하면 활성화된 플러그인을 호출하여 검색 결과를 반환하고, 선택 시 제목/작가/소개글/표지를 실시간 오버라이트 반영합니다.

## 4. 보안 설계, 개별 카테고리 권한 통제 및 OPDS 인증 규격

### 💡 개요
서재 데이터를 유출당하지 않기 위해 외부 침입으로부터의 차단 수단을 마련하고 모바일 외부 뷰어 기기들을 위한 규격을 준수합니다.

### 🛠️ 구현 메커니즘
* **권한 분리 및 격리**:
  - 세션 기반 인증 필터(`login_required`)가 작동하여 로그인 없이 어떤 미디어 API로도 다이렉트 호출할 수 없습니다.
  - 성인 책의 이미지 스트리밍 엔드포인트는 로그인된 유저의 `has_adult_access` 레코드를 세션과 대조 검증하여 권한이 없는 경우 `403 Forbidden`을 리턴합니다.
* **[NEW] 카테고리별 개별 권한 제어**:
  - 관리자 권한으로 계정별 접근 가능한 라이브러리 카테고리를 테이블 상에서 매핑하여 즉시 허용/차단할 수 있는 `user_category_permissions` 스키마 및 UI 제어 흐름이 구현되었습니다.
* **OPDS (Open Publication Distribution System) 표준 규격 및 리더 앱 연동**:
  - 모바일 리더 앱과의 호환을 위한 OPDS XML 피드 API(`/opds`, `/opds-adult`)를 지원합니다.
  - 특히 **타치요미(Tachiyomi), 미혼(Mihon)** 등의 전용 OPDS 연동 스펙에 맞추어 설계된 **`/app-opds`** 및 **`/app-opds-adult`** 엔드포인트를 제공하여 모바일 서재 다운로드와 스트리밍 연동의 안정성을 높였습니다.
  - 외부 기기 연동 시 보안 통신을 보장하기 위해 **HTTP Basic Authentication**을 거치며, 데이터베이스에 등록된 유저 크레덴셜 정보와 동적으로 암호 대조를 실행합니다.

## 5. [NEW] 모바일 최적화 및 견고한 4계층 아키텍처

### 💡 개요
모바일/태블릿 등 다양한 폼팩터 기기에서의 무결한 렌더링 성능과 대용량 서비스 확장을 위한 백엔드 레이어 분리 구조입니다.

### 🛠️ 구현 메커니즘
* **노치 및 상단/하단 안전 영역 대응 (Safe Area Insets)**:
  - iOS 단말(iPhone, iPad) 등 가변 노치나 하단 스와이프 바가 존재하는 환경에서 렌더링 영역이 관통되거나 툴바에 밀려 잘리는 오차를 제거하도록, CSS `env(safe-area-inset-top)` 및 `env(safe-area-inset-bottom)`을 연동하고 웹킷(WebKit) 렌더러에서의 높이 유실을 막기 위해 absolute 높이를 `height: 100% !important;`로 물리 격리 처리했습니다.
* **다국어(i18n) innerHTML 안전 렌더링**:
  - 다국어 번역 리소스 내에 `<strong>` 등 마크업 태그가 포함되어 있을 시 텍스트 코드가 그대로 노출되던 문제를 감지하여, 번역 대상에 돔 태그가 있는 경우 자동으로 `innerHTML`을 활용하여 유려하게 렌더링해 주는 헬퍼를 이식했습니다.
* **비즈니스-저장소 레이어 격리 (4-Tier Architecture)**:
  - 쿼리가 하드코딩되던 백엔드를 `UserRepository`, `CategoryRepository`, `SettingsRepository` 등의 레포지토리 패턴 레이어로 완벽 격리하여 데이터 인프라(SQLite ➡️ PostgreSQL 등)의 이관 유연성을 대폭 확장했습니다.
  - SQLite 데이터베이스의 `REINDEX`, `VACUUM` 물리 파편화 압축 정리를 서비스 모듈인 `db_tuning_service.py`로 격리하여 시스템 관리 연산 결합도를 안전하게 억제했습니다.

## 6. [NEW] 사용자 커스텀 단축키 및 전역 콘솔 로그 제어

### 💡 개요
리눅스 등 OS 데스크톱 고유 단축키 충돌을 회피하기 위해 전역 단축키를 유저가 실시간 커스터마이징하고, 운영 환경에서 번잡한 브라우저 디버깅 메시지를 일괄 소거할 수 있는 관리 스펙입니다.

### 🛠️ 구현 메커니즘
* **브라우저 keydown 단축키 레코더 및 로컬스토리지 적재**:
  - 일반설정 탭(`general_tab.html`) 내에 키 조합 레코더 UI를 심어, 사용자가 누르는 키 이벤트를 가로채 modifiers 조합키와 일반 입력 키를 디스플레이 문자열로 포맷한 뒤 로컬스토리지에 저장합니다.
* **실시간 keydown 디텍션 구조**:
  - 모듈 스코프 유실과 렌더링 지연을 예방하기 위해, 이벤트 리스너 내에서 keydown 감지 찰나에 로컬스토리지를 다이렉트 디코딩 매칭하는 아키텍처를 도입하여 새로고침 없이 즉석에서 바뀐 단축키가 동작하도록 설계했습니다.
* **VIEW_LOG 환경변수 연동 프론트엔드 전역 몽키 패치**:
  - `.env`에 등록된 `VIEW_LOG` 변수를 Flask가 파싱하여 웹 렌더링 주소로 노출하며, 그 값이 `true`가 아닐 때 index.html 로딩 시점에 `console.log` 및 `console.warn` 객체를 빈 함수(Null-Function)로 덮어씌워 브라우저 로깅 과부하 및 콘솔 가독성을 대폭 최적화했습니다. (안전상의 이유로 console.error는 살려둠)
