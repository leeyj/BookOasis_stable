# 📊 미디어 서버 비교 분석: Kavita vs Komga vs Stump vs BookOasis

이 문서는 대표적인 오픈소스 만화/도서 미디어 서버인 **Kavita**, **Komga**, **Stump**, 그리고 초경량 아키텍처로 개발된 **BookOasis**의 기능, 성능, 도서 관리 능력을 객관적으로 비교한 자료입니다.

## 1. 핵심 아키텍처 및 철학 비교

| 항목 | Kavita | Komga | Stump | BookOasis |
| :--- | :--- | :--- | :--- | :--- |
| **개발 언어** | C# (.NET) + Angular | Java (Spring Boot) + Vue.js | Rust (Axum) + React | Python (Flask) + Vanilla JS |
| **의존성/무게** | 무거움 (방대한 프레임워크) | 매우 무거움 (JVM 환경 필수) | 보통 (Rust 기반 컴파일러 네이티브) | **초경량** (노빌드, 서드파티 최소화) |
| **스트리밍 방식** | 캐시 기반 압축 해제 및 전송 | 캐시 기반 전처리 및 전송 | 캐시/임시 파일 압축 해제 전송 | **오프셋 기반 실시간 다이렉트 전송** |
| **설치 편의성** | 보통 (Docker 권장) | 보통 (Docker 권장) | 보통 (Docker/Native Binary) | **최상** (Native Python만으로도 1분 컷) |

---

## 2. 성능 (Performance) 비교

BookOasis는 극단적인 최적화를 목표로 설계되었기 때문에 리소스 사용량과 체감 속도에서 압도적인 우위를 가집니다.

| 항목 | Kavita | Komga | Stump | BookOasis |
| :--- | :--- | :--- | :--- | :--- |
| **디스크 I/O** | 높음 (스캔/읽기 시 압축 해제 잦음) | 매우 높음 (백그라운드 캐싱 빈번) | 보통 (파일 단위 분석 및 추출) | **최저** (물리적 압축 해제 100% 배제) |
| **RAM 점유율** | 높음 (500MB ~ 1GB 이상) | 매우 높음 (1GB ~ 2GB 이상) | 보통 (100MB ~ 300MB 수준) | **초저전력** (수십 MB 수준 유지) |
| **뷰어 로딩 속도**| 보통 (캐시가 없을 시 지연 발생) | 보통 (초기 로딩 시 딜레이 존재) | 보통 (자체 디코딩 지연 최소화) | **즉각적 (Zero-Delay)** |
| **대용량 스캔** | 다소 느림 (메타데이터 심층 파싱) | 다소 느림 (정교한 분석 수행) | 보통 (Rust로 빠르게 인덱싱) | **매우 빠름** (필수 정보 및 오프셋만 고속 추출) |

> [!TIP]
> **성능 총평**: **BookOasis**가 라즈베리파이나 저사양 NAS 등에서 가장 높은 효율을 발휘합니다. **Stump**는 Rust로 구현되어 JVM/Dotnet 기반인 Komga/Kavita에 비해서는 가볍고 빠르지만, 여전히 파일 구조 전체 분석 및 캐싱 레이어를 거치므로 BookOasis의 오프셋 기반 실시간 전송과 극도로 가벼운 구조에는 미치지 못합니다.

## 3. 기능 (Features) 비교

| 항목 | Kavita | Komga | Stump | BookOasis |
| :--- | :--- | :--- | :--- | :--- |
| **지원 포맷** | ZIP, CBZ, CBR, EPUB, PDF 등 | ZIP, CBZ, CBR, PDF, EPUB 등 | ZIP, CBZ, CBR, PDF, EPUB | **ZIP, CBZ, PDF, TXT** |
| **서드파티 연동** | OPDS, 다양한 외부 리더기 완벽 지원 | OPDS, Tachiyomi 등 생태계 방대 | OPDS (Tachiyomi 등 호환) | OPDS 지원 (Tachiyomi 등 연동 가능) |
| **UI/UX** | 매우 화려하고 현대적 | 깔끔하고 정돈된 디자인 | 모던 넷플릭스 스타일 UI (React) | 미니멀, 모바일 최적화 (Vanilla JS) |
| **플러그인 확장** | 자체 내장 로직 의존 | 자체 내장 로직 의존 | 자체 내장 로직 의존 | **웹 UI 기반 동적 플러그인 아키텍처** |
| **성인/권한 제어**| 지원 (계정별 라이브러리 할당) | 지원 (세밀한 권한 제어 가능) | 지원 (상세 권한 및 사용자 제어) | 지원 (계정별 성인 인증 플래그 격리) |

> [!NOTE]
> **기능 총평**: 다중 포맷 지원 및 화려한 전용 UI/앱 생태계 측면에서는 **Kavita**, **Komga**, **Stump**가 풍부합니다. 반면 **BookOasis**는 텍스트(TXT) 지원이 내장되어 있어 국내 웹소설 독자층에 강점을 가지며, 동적 플러그인 아키텍처를 통해 서버 리스타트 없이 플러그인을 활성화할 수 있는 독자적인 매커니즘을 제공합니다.

## 4. 도서 관리 능력 (Book Management) 비교

| 항목 | Kavita | Komga | Stump | BookOasis |
| :--- | :--- | :--- | :--- | :--- |
| **자동 메타데이터**| ComicInfo.xml, EPUB 메타 자동 파싱 | ComicInfo.xml 파싱 매우 강력함 | ComicInfo.xml 기본 파싱 지원 | **ComicInfo.xml 기본 파싱 지원** + 외부 API(알라딘 등) 플러그인 연동 |
| **시리즈 그룹화** | **최상** (이름 규칙 및 메타로 완벽히 묶음) | **최상** (컬렉션, 시리즈 관리 탁월) | 보통 (메타데이터 및 디렉터리 기반) | 보통 (디렉터리/폴더 구조에 주로 의존) |
| **에러 파일 처리** | 스캔 중단되거나 묵시적 무시 | 에러 로그 남김 | 에러 로그 기록 | **전용 '에러 리포트' 아키텍처 아카이빙** |
| **한국어 친화성** | 다국어 지원하나 한국 메타 부족 | 다국어 지원하나 한국 메타 부족 | 다국어 지원하나 한국 메타 부족 | **최상 (국내 전용 알라딘 API 완벽 연동)** |

> [!IMPORTANT]
> **관리 능력 총평**:
> * **Kavita / Komga / Stump**: 서양 코믹스의 표준인 `ComicInfo.xml` 및 EPUB 구조를 적극 활용하여 시리즈 그룹화 및 컬렉션 구축이 견고합니다.
> * **BookOasis**: 파일 포맷 구조에 크게 구애받지 않도록 물리적 폴더 구조를 직관적으로 반영하며, 국내 독자들을 위해 **알라딘 API 플러그인**을 장착하여 한글 메타데이터 및 책 표지 이미지를 원클릭으로 보강하는 강력한 한국어 특화 이점이 있습니다.

## 💡 결론 요약 (Conclusion)

1. **Komga**: 가장 널리 쓰이며 안정적인 **표준적 만화 미디어 서버**. (서양 코믹스 중심)
2. **Kavita**: EPUB 소설과 만화를 아우르는 **화려하고 현대적인 올라운더**.
3. **Stump**: Rust 기반의 컴팩트함과 넷플릭스 스타일 UI를 강조한 **차세대 신생 서버**.
4. **BookOasis**: 극도의 가벼움, 무설치에 가까운 편리함, 그리고 텍스트(TXT) 및 한국어 책(알라딘 API) 메타데이터 연동에 특화된 **실전 지향형 초경량 미디어 서버**.
